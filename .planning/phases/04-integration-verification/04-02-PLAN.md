If the log grep finds `<anylinuxfs-type:ext4>` or similar in the log output, a tag is being emitted at runtime — the vmproxy binary was not rebuilt after Phase 2, or cmd_mount.rs is parsing stdout instead of IPC. <threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| host process→hypervisor | anylinuxfs invokes libkrun to create a microVM; hypervisor entitlement required |
| host→guest (IPC socket) | VmEvent messages flow from vmproxy over vsock-mapped Unix socket |
| test scripts→host binary | BATS invokes bin/anylinuxfs with test-constructed disk images |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-05 | Tampering | test disk images | accept | BATS uses pre-built test images in the repo; images are not user-supplied during normal testing; low risk in dev context |
| T-04-06 | Information Disclosure | anylinuxfs log file | mitigate | QUAL-03 runtime check: grep log for `<anylinuxfs-` excluding `vmproxy-ready`; any match means structured VM data leaked to the log (regression) |
| T-04-07 | Denial of Service | concurrent test mounts | accept | anylinuxfs serializes concurrent mounts via /tmp/anylinuxfs.lock; bats runs sequentially by default; no parallelism risk |
| T-04-08 | Spoofing | stale binary | mitigate | Task 1 verifies binary freshness by checking mtime vs source; if stale, build-app.sh is run to produce a binary that includes all Phase 2+3 changes |
| T-04-09 | Elevation of Privilege | build-app.sh codesign | accept | build-app.sh runs `codesign --sign -` (ad-hoc signing) on bin/anylinuxfs; this is the established dev workflow, no certificate required |
</threat_model>
